Provisioning

Interview once, deploy the right tier

Before writing a single file, Claunchpad runs a short interview: it auto-detects expertise signals from the repo (existing .claude/ directory, git depth, CI config), asks one confirming question (New / Comfortable / Advanced), then probes project shape — size, nature, stakes, collaboration, and agent need. The result is a tier recommendation the user can override.

The Three Pillars

Memory · Delegation · Orchestration

Every tier provisions these three pillars in the depth the project actually needs.

💾

Memory

Durable project knowledge that survives /clear, compaction, and subagent handoffs.

  • Starter: single MEMORY.md — Decisions / Learnings / Errors / References sections
  • Graduated: per-fact memory/ directory; MEMORY.md becomes a thin index
  • Fact files carry frontmatter (name, description, type, date) and wikilinks
  • load-memory hook injects both shapes via additionalContext each session
  • General-purpose subagents inherit via CLAUDE.md; Explore/Plan need explicit injection
📋

Delegation

Vague delegation is the top cause of duplicated work and gaps. The 7-point contract removes ambiguity.

  • All seven points must appear in every non-trivial delegation prompt
  • General-purpose subagents inherit CLAUDE.md — don't re-explain conventions
  • Explore/Plan agents skip CLAUDE.md — inject all constraints explicitly
  • No subagent inherits parent conversation or memory stores
  • Nesting bounded: ~5 levels deep loses the Agent tool
  • Return-learnings slot feeds findings back into project memory
📈

Orchestration

Solo → Pair → Parallel fan-out → Workflow. Escalate one tier only on a concrete trigger.

  • ~15x token cost for large multi-agent systems (Anthropic research)
  • Helps on breadth-first, independent, parallelizable work
  • Hurts on tightly-coupled work with shared state or same-file edits
  • Efficiency governor: no fired trigger = no escalation
  • Synthesis: cross-check results, attribute sources, take a position
7-Point Delegation Contract
01
Objective
One concrete outcome in one sentence — the thing that must be true when the agent stops.
02
Scope
What's in, what's out, what not to touch. Prevents scope creep and unauthorized file edits.
03
Context
Relevant memory facts pasted directly, plus file paths and error text. The agent cannot read memory on its own.
04
Output Format
Exactly what to return — JSON, bullet list, diff, verdict — so the result composes without rework.
05
Tool Guidance
Which tools to prefer or avoid; fire independent calls in parallel when sub-tasks don't share state.
06
Stop Criteria
When the task is done. Prevents runaway work and unnecessary continuation past the objective.
07
Return-Learnings Slot
Required block: FAILURE / CAUSE / FIX and LEARNING items that feed back into memory after the run.
Role Model Characteristic work
Mechanical / high-volume Haiku Formatting, log triage, simple lookups, classification
Implementation / search / review Sonnet Writing code, searching a codebase, reviewing a change
Hard reasoning / synthesis / judging Opus Architecture decisions, combining many results, adversarial review, hardest bugs
Hook Layer

Automate loading; nudge harvesting

Two POSIX sh hooks wire the memory lifecycle. They live in .claude/hooks/ and are registered in .claude/settings.json at provisioning time.

SessionStart load-memory

Runs at the start of every session — including after /clear and after compaction (matched via startup|resume|clear|compact).

  • Reads MEMORY.md if it exists; emits it as additionalContext
  • Reads up to 8 most-recently-modified memory/*.md files (graduated projects)
  • On compact source: reloads .claunchpad/handoff.md buffer (Pro tier)
  • Clears the per-session harvest-nudge flag so nudge fires again
  • Exits silently if nothing to load — no noise on empty projects

Deployed at all three tiers. Memory loads without depending on Claude remembering to read it.

Stop harvest-nudge

Runs at the end of every turn (Standard and Pro tiers). Fires once per session, then steps aside.

  • Checks for .claunchpad/.harvest-nudged flag; if present, exits silently
  • If absent: creates the flag, then blocks the stop with a reminder
  • Reminder: "file any durable decision, resolved failure, or technique — or say there's nothing worth keeping"
  • Never writes to memory itself — always a deliberate human choice
  • Flag is cleared by load-memory at the next session start

Backs up any forgotten harvest at session end. Writing is never automated.

"Load auto, harvest nudge" — automation loads memory reliably; writing is always under deliberate control.
Orchestration

The 4-tier front door

Start here every time. Escalate one tier only when a concrete trigger fires. De-escalate the moment a cheaper structure would finish correctly.

Tier 1 — Solo
Default. Task is trivial, sequential, or tightly-coupled to the same file. Parallelism would cause conflicts or add no speed. Most tasks live here.
Tier 2 — Pair
Default for real changes. One agent builds; a separate fresh-context agent reviews; orchestrator verifies and merges. Trigger: the change touches shared logic or warrants a second pair of eyes.
Tier 3 — Parallel fan-out
3+ genuinely independent angles — subtasks that don't share state and won't edit the same files (research pod, review board, best-of-N). Do not use if agents would touch the same files; last write wins.
Tier 4 — Workflow
10+ uniform targets (files to migrate, endpoints to audit) or a multi-module program where hand-spawning every agent is impractical. Hand the work to a deterministic script that orchestrates many agents.
The ~15x cost reality. Anthropic's multi-agent research measured roughly 15x the tokens of a single chat for large systems. Multi-agent helps on breadth-first, independent, parallelizable work. It hurts on tightly-coupled work with many dependencies. The question is never "how many agents?" — it's "what is the smallest formation that produces a correct, verified result?"
Councils — Application Layer

Structured pressure-tests on top of the OS

Councils are one application built on Claunchpad — they use the memory pillar for framing and findings, and the orchestration pillar for parallel lenses. Pro tier wires council presets into CLAUDE.md; Standard mentions them on request; Starter hides them unless asked.

Decision Council

Consequential, hard-to-reverse choices

Claude is agreeable by default — ask if an idea is good and it argues yes; ask if it's bad and it argues that too. The Decision Council forces structured disagreement across five adversarial advisors and ends with the truth you're most likely resisting.

Convene before any consequential, hard-to-reverse commitment: launching something public, hiring/firing, pivoting or killing a project, large or recurring spend, irreversible architecture choice, signing a contract. Not for low-stakes or reversible choices — deliberately heavy.

Procedure: Frame → Five testimonies (pure agreement forbidden) → Anonymous peer review (advisors critique each other without knowing who said what) → Chair's verdict: Proceed / Proceed with conditions / Rework first / Don't — plus "The hardest truth" the user most needs to hear.

Contrarian First-Principles Analyst Optimist/Expansionist Outsider Executor

Feasibility Council

Whole-project viability review

Where the Decision Council pressure-tests a single irreversible choice with adversarial roles, the Feasibility Council evaluates whether an entire initiative can actually work — across the functional dimensions that decide its fate.

Convene when committing real resources to a project: greenlighting a new product, evaluating a venture for go/no-go, or a health check on something in flight. Not for a single reversible call or a project still at the sentence stage.

Procedure: Frame → Four lens assessments (Green/Amber/Red per dimension) → Cross-examination (where do the plans contradict each other?) → Chair's verdict gated by the weakest link — plus "The thing that has to be true" across all four lenses.

Technology/Build Finance/Economics Go-to-Market Marketing/Positioning

Feasibility is set by the worst dimension, not the average. Green on three lenses but Red on distribution is a Red project.

Recommended Skills — Bootstrap Offer

Offer to install the ecosystem, not just your scaffold

After deploying your tier, Claunchpad offers a curated set of high-value Claude Code plugins/skills: [Y] install all · [c] customise · [s] skip. On Claude Code it runs each skill's own install command (they differ — there is no uniform installer); on claude.ai, where there is no plugin system, it prints a manual list instead. The set lives in an editable manifest, so it is easy to change.

Superpowers

Skill-engineering backbone

Brainstorming, writing-plans, TDD, systematic-debugging, subagent-driven-development — the disciplined workflow engine (the same one used to build this project).

obra/superpowers-marketplace claude plugin install

Skills Curator

Evaluate before you install

Scores a candidate skill before you add it and persists every decision — "decide once, re-decide never." The natural gatekeeper for everything else you install.

captkernel/Skills_Curator clone + install.sh / .ps1

Agent Browser

Browser automation for agents

Lets agents drive a real browser — tab scan/switch, JS eval, screenshots, uploads. Heaviest of the three: needs Node and downloads a Chrome build on first run.

vercel-labs/agent-browser npm i -g + agent-browser install

Install is never fully silent — each skill skips gracefully if its prerequisite is missing, and Claunchpad always tells you the one human step left: restart Claude Code (or run /reload-plugins) to activate.

Without Claunchpad

You are the memory, the reviewer, and the setup.
  1. New session: Claude forgot yesterday. You re-explain the stack, the plan, and the gotcha that cost you two hours — again.
  2. Big change: done in one solo pass with no review. A subtle bug ships.
  3. You type /clear to free up context. Everything it learned is gone.
  4. You want browser automation: you hunt for a skill, install it by hand, hope it fits.
  5. Tomorrow: back to step 1. The project never compounds.
VS

With Claunchpad

The project remembers, reviews, and equips itself.
  1. New session: the load-memory hook reloads yesterday's decisions, fixes, and gotchas automatically — Claude picks up mid-stride.
  2. Big change: it runs as a Pair — build, independent review, verify. The bug is caught before merge.
  3. You type /clear. Memory lives on disk; the next session reloads it. Nothing lost.
  4. Browser automation was offered and installed at setup. It's already there.
  5. Tomorrow: it starts smarter than today. The project compounds.

Why trust it

Built the way it preaches

Claunchpad was built using its own discipline — so the rigor it asks of you is rigor it survived first.

Every piece was implemented by a fresh agent, reviewed by a second one for correctness and for building only what was asked, and verified against an automated test suite before it counted as done. At the end, a single broad review read the whole thing at once — and immediately earned its keep, catching two bugs every narrow per-task review had structurally missed: a feature the docs proudly described but no code actually performed, and a brand-new file quietly pointing at two files the same change had just deleted. Both were fixed before anything shipped.

A tool that cuts corners in its own construction has no business asking you to trust its discipline. This one doesn't cut them.
